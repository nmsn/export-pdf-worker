# `src/lib/pdf-worker.ts` 执行历程梳理

这份文档说明 [src/lib/pdf-worker.ts](src/lib/pdf-worker.ts) 当前实现的完整执行过程，以及这套方案在演进过程中试图解决的问题、关键难点和设计亮点。

## 1. 这份文件要解决什么问题

传统 PDF 导出最大的问题不是 `jsPDF` 本身，而是“图片加载 + 图片解码 + base64 转换 + PDF 绘制”都堆在主线程上。

当导出内容里有多张大图时，会出现几类明显问题：

- 主线程长时间阻塞，页面交互发卡。
- 图片数据体积大，序列化和传输成本高。
- 多页 PDF 的每一页本质上可以独立渲染，但传统方案没有真正把并行能力用起来。
- 目录、分页、页眉、图表编号这些排版逻辑不能丢，不能为了性能牺牲功能完整性。

因此，`pdf-worker.ts` 的目标不是“把 PDF 放到 Worker 里”这么简单，而是把整个导出过程拆成两个阶段：

1. 主线程只负责排版和收集绘制指令。
2. Worker 负责真正的页面渲染，最后再把每页 PDF 合并成一个文件。

## 2. 整体设计思路

当前实现的核心思想是“先布局，后渲染”。

### 2.1 主线程负责什么

主线程并不直接往最终 PDF 上画内容，而是做这些事：

- 维护分页状态、章节编号、图表编号。
- 根据文本、图片、表格计算布局位置。
- 把每一页内容收集成 `DrawInstructionV2[]` 指令数组。
- 预加载图片，拿到尺寸和可复用的数据。
- 把每一页指令分发给多个 Worker 并行渲染。

### 2.2 Worker 负责什么

Worker 只做一件事：拿到一页的绘制指令，创建一个独立的 `jsPDF` 实例，画完这页并返回 `ArrayBuffer`。

也就是说，Worker 不共享同一个 `doc`，而是“一页一个 PDF”。最后由主线程使用 `pdf-lib` 统一合并。

这个拆分非常关键，因为 `jsPDF` 的文档实例本质上是有状态对象，不适合跨线程共享。

## 3. 入口执行链路

对外入口是 [src/lib/pdf-worker.ts](src/lib/pdf-worker.ts#L1720) 里的 `exportPdfWithWorker`。

它的执行顺序可以概括成下面这 8 步：

1. 检测浏览器是否支持 `OffscreenCanvas` 和 `createImageBitmap`。
2. 初始化 `PDFWorkerV2` 实例。
3. 遍历业务数据，把标题、正文、图片、表格转换成页面指令。
4. 在第一页插入目录页指令。
5. 预加载所有注册过的图片。
6. 把每一页指令分发给 Worker 池并行渲染。
7. 用 `pdf-lib` 按顺序合并所有页面二进制。
8. 生成 Blob，触发浏览器下载。

## 4. 分阶段执行过程

### 4.1 阶段一：能力检测

文件最开始先做浏览器能力判断，见 [src/lib/pdf-worker.ts](src/lib/pdf-worker.ts#L9)。

核心判断只有两个：

- `OffscreenCanvas`
- `createImageBitmap`

只有两者同时可用时，才走优化路径 `SUPPORTS_WORKER_OPTIMIZATION`。否则进入降级方案。

这一步很重要，因为后面的图片预处理和 Worker 内绘图都依赖这些浏览器能力。

### 4.2 阶段二：初始化排版引擎

`PDFWorkerV2` 在 [src/lib/pdf-worker.ts](src/lib/pdf-worker.ts#L908) 定义。

构造阶段主要初始化这些状态：

- 页面尺寸、边距、内边距。
- 当前写入坐标 `x/y`。
- 章节序号栈 `serialStack`。
- 当前页指令 `currentPageInstructions`。
- 所有页面指令 `allInstructions`。
- 图片 URL、图片尺寸信息、图片索引。
- 可选的图片 Worker Pool。

这里有一个关键点：构造函数里创建了一个临时 `JsPdf` 实例，但目的不是导出，而是读取页面宽高用于布局计算。

### 4.3 阶段三：把业务数据变成“页面指令”

`exportPdfWithWorker` 会遍历外部传入的结构化数据，在 [src/lib/pdf-worker.ts](src/lib/pdf-worker.ts#L1744) 到 [src/lib/pdf-worker.ts](src/lib/pdf-worker.ts#L1772) 之间依次调用：

- `addChapter`
- `addPage`
- `addTable`
- `addImage`
- `addText`

这一步不是生成 PDF，而是在做“离线排版”。

#### 文本

文本由 `collectTextInstructions` 计算真实坐标、换行高度、缩进和对齐方式，最后产出一组 `TextInstruction`。

#### 图片

图片先通过 `registerImage` 分配全局 `imageIndex`，然后用 `collectImageInstructionsV2` 结合图片宽高、当前剩余空间、页高、底部描述等信息，决定：

- 当前页是否放得下。
- 是否需要缩放。
- 是否需要换页。
- 最终放在什么位置。

#### 表格

表格在布局阶段只记录表头、表体、样式和 `startY`，真正的 `autoTable` 渲染在 Worker 内执行。

#### 章节和目录

章节编号由 `createSerialStack` 管理，目录页由 `addCatalog` 统一插入到指定页。

这意味着当前实现不是“边生成边出结果”，而是先完整收集一遍结构，再渲染。

### 4.4 阶段四：图片预加载

图片预加载由 `preloadImages` 完成，位置在 [src/lib/pdf-worker.ts](src/lib/pdf-worker.ts#L981)。

这里存在两条路径：

- 优化路径：用 `ImagePreloader` 并行拉取图片、解码图片，并统一转换成 JPEG `Uint8Array`。
- 降级路径：主线程直接加载图片，只记录尺寸信息，后面再转 base64。

这一步的意义有两个：

1. 提前知道图片真实宽高，布局才能稳定。
2. 把昂贵的图片处理尽量前移，并从主线程挪走。

### 4.5 阶段五：拿到最终页面指令

`getPageInstructions` 会把当前正在编辑的最后一页也推入 `allInstructions`，然后把结构整理成：

```ts
{ pageIndex, items }
```

每个 `items` 都是一页完整的绘制描述。

这一步做完之后，整份文档已经从“业务数据”转成“渲染数据”了。

### 4.6 阶段六：Worker 并行渲染

这部分是整个文件最核心的执行过程，对应 [src/lib/pdf-worker.ts](src/lib/pdf-worker.ts#L1351)。

#### 优化路径

支持优化能力时，流程是：

1. 创建 PDF 渲染 Worker 池。
2. 为每一页收集该页真正用到的图片索引。
3. 从 `imagePreloader` 缓存中取出对应 `Uint8Array`。
4. 调用 Worker 内的 `renderPage`。
5. Worker 内重新创建 `jsPDF`，依次绘制文本、图片、表格。
6. 当前页绘制完成后返回 `ArrayBuffer`。

#### 降级路径

不支持优化能力时：

1. 主线程先把所有图片转成 base64。
2. 再创建降级版 Worker 池。
3. 每一页通过 `renderPageWithBase64` 渲染。

两条路径的共同点是：

- 都是“按页并行”。
- 都是“一页一个 `jsPDF` 实例”。
- 最终返回的都是独立页面二进制。

### 4.7 阶段七：合并 PDF

所有页渲染完成后，`mergePDFs` 会在 [src/lib/pdf-worker.ts](src/lib/pdf-worker.ts#L1597) 用 `pdf-lib` 顺序载入每个页面 PDF，再复制到最终文档里。

这个阶段不是最耗 CPU 的，但它是并行方案成立的必要收口步骤。

因为前面选择的是“单页独立渲染”，后面就必须有一个统一合并器。

### 4.8 阶段八：下载与资源释放

`save` 在 [src/lib/pdf-worker.ts](src/lib/pdf-worker.ts#L1613) 做最后的收尾：

- 调用 `renderWithWorkers`
- 调用 `mergePDFs`
- 释放 `imageWorkerPool`
- 释放 `imagePreloader`
- 生成 Blob URL
- 触发 `<a>` 下载
- 打印 timing 汇总

这里的资源释放不能省，因为这份实现创建了动态 Worker 和对象 URL。

## 5. 中间最难的几个点

### 5.1 难点一：`jsPDF` 不是天然适合多线程共享

这是整个方案的根问题。

`jsPDF` 的 `doc` 是有状态的，里面包含：

- 当前页状态
- 当前字体状态
- 当前绘制缓冲区
- 插件状态，比如 `autoTable`

这类对象既不适合 `structuredClone`，也不适合跨线程共享。

所以真正可行的方式不是“把同一个 doc 扔给 Worker”，而是：

- 主线程负责布局
- Worker 负责单页渲染
- 主线程最后合并

这是整个架构从“线程搬运”变成“渲染分片”的关键转折。

### 5.2 难点二：图片才是瓶颈，不是文本

一开始很容易误判为“PDF 渲染慢”，但实际最重的是图片链路：

- fetch
- decode
- createImageBitmap
- canvas 转码
- base64 转换

如果这些还在主线程，即便渲染动作放到 Worker，整体也只是伪并行。

因此这份实现的真正优化重点落在图片预加载和图片数据传递，而不是单纯把 `doc.text()` 挪到 Worker。

### 5.3 难点三：图片既要参与布局，又要参与渲染

图片处理有一个天然矛盾：

- 布局时只需要尺寸。
- 渲染时需要实际像素数据。

因此当前实现把图片管理拆成两个维度：

- `imageInfos` 存尺寸
- `imagePreloader` 存实际二进制数据

这样布局和渲染可以共享同一套图片索引，但不强耦合在一个对象上。

### 5.4 难点四：分页逻辑不能因为 Worker 化而失真

并行渲染并不意味着分页就简单了。

图片和文本都可能触发分页，尤其是图片：

- 图片过大，超过单页总高度。
- 图片在当前页剩余空间不足。
- 图片可按比例缩小留在本页，还是应该整张挪到下一页。

`collectImageInstructionsV2` 的价值在于，它把分页判断留在主线程布局阶段完成，而不是让 Worker 在渲染时临场决定。

否则会出现布局和目录页码不一致的问题。

### 5.5 难点五：目录页插入会影响页码一致性

目录并不是最后单独附加，而是插入到指定页位置，默认是第一页。

这意味着章节记录、目录项、真实页码之间要保持一致。

当前实现先记录章节，再统一 `addCatalog(1)`，本质上是在“页面结构已经稳定之后”再做目录注入。

这是一个比较务实的做法。

### 5.6 难点六：浏览器兼容性不能靠理想环境假设

`OffscreenCanvas` 和 `createImageBitmap` 并不是所有浏览器环境都完整支持。

如果代码只实现理想路径，最终用户体验会非常脆弱。

所以文件一开始就做能力检测，并保留完整降级流程。这不是锦上添花，而是生产可用性的基础。

## 6. 这份实现的亮点

### 6.1 亮点一：把“导出”拆成布局层和渲染层

这是最有价值的结构化改造。

从结果看，`pdf-worker.ts` 做的不是“多线程版 jsPDF”，而是：

- 布局引擎
- 指令协议
- Worker 渲染层
- 合并层

这让后续优化有了明确边界。

### 6.2 亮点二：指令化设计降低了线程耦合

`DrawInstructionV2` 把文本、图片、表格统一抽象成纯数据。

这个抽象带来几个直接收益：

- 主线程和 Worker 的职责边界很清晰。
- 更容易调试单页问题。
- 以后如果替换渲染后端，改动点更集中。

### 6.3 亮点三：按页并行是正确的并行粒度

并行粒度太粗，收益不明显；太细，调度开销高。

按页并行正好处在一个合理平衡点：

- 页面天然独立。
- 页内顺序仍然受控。
- 合并步骤清晰。
- Worker 数量也比较容易按 CPU 核数调优。

### 6.4 亮点四：图片预处理前移

把图片处理从渲染阶段前移到预加载阶段，直接改善了关键路径时延。

这比单纯“渲染更快”更重要，因为它减少了用户点击导出之后的阻塞感。

### 6.5 亮点五：保留了功能完整性

这份实现不是只追求 benchmark 数字，还保留了：

- 页眉
- 章节编号
- 图号/表号
- 目录页
- 分页逻辑
- 图片说明文案

这说明它不是一个 demo 级别的并行尝试，而是在认真替换原有导出能力。

## 7. 当前实现里的现实取舍

这份代码并不是“理论最优解”，而是工程上可落地的折中版本。

几个明显的取舍包括：

- Worker 代码以内联字符串方式创建，开发效率高，但维护成本偏高。
- 优化路径里仍然把 `Uint8Array` 转成 base64 再喂给 `jsPDF`，没有做到绝对意义上的最小拷贝。
- 目录逻辑有一定重复构造痕迹，说明它是在功能补齐过程中逐步长出来的。
- 构造布局时会多次创建临时 `JsPdf` 实例，结构上清晰，但仍有进一步收敛空间。

这些取舍不代表方案失败，反而说明它优先解决了真正影响体验的问题。

## 8. 一句话总结

`src/lib/pdf-worker.ts` 的本质不是“把 PDF 导出搬进 Worker”，而是把 PDF 导出改造成一条分阶段流水线：

- 主线程做布局
- Worker 做单页渲染
- 主线程做合并和下载

真正的难点在图片、分页和状态切分；真正的亮点在指令化建模、按页并行和兼容性降级。

如果后续要继续演进，这份文件最值得继续拆分的方向有两个：

- 把内联 Worker 代码抽到独立 Worker 文件。
- 把布局工具、图片预处理、渲染协议拆成独立模块。
